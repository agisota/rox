## Краткий обзор

```text
rox <command> [subcommand] [options]
```

Запустите `rox --help` для списка команд верхнего уровня или
`rox <command> --help` для любой группы.

## Глобальные опции

Эти флаги принимаются каждой командой:

| Флаг | Env | Описание |
| --- | --- | --- |
| `--json` | | Вывести полезную нагрузку данных в виде форматированного JSON. |
| `--quiet` | | По одному ID на строку для массивов; ID для одиночных объектов; иначе откат к JSON. |
| `--api-key <key>` | `ROX_API_KEY` | Использовать API-ключ вместо сохранённого OAuth-входа. |
| `--help`, `-h` | | Показать справку для текущей команды. |
| `--version`, `-v` | | Вывести `rox v<version>` и выйти. |

## Команды

### auth

Аутентификация и просмотр сессии.

<Command
	name="rox auth login"
	options={[
		{
			flag: "--organization <idOrSlug>",
			description: "Выбирает активную организацию без запроса. Требуется для входов без TTY, когда вы состоите в нескольких организациях.",
		},
		{
			flag: "--api-key <key>",
			description: "Сохранить API-ключ Rox (`sk_live_…`) в `~/.rox/config.json` вместо запуска OAuth-потока.",
		},
	]}
>
Аутентификация через браузерный OAuth и сохранение токена сессии в
`~/.rox/config.json`.

Аккаунты с одной организацией выбираются автоматически. При нескольких организациях и наличии
TTY CLI выводит запрос; при нескольких организациях и отсутствии TTY вы должны передать
`--organization`, иначе команда завершается с кодом 1, выведя список доступных slug.

```bash
rox auth login
rox auth login --organization acme
```

Чтобы войти с API-ключом — полезно для CI или любого окружения, где
браузерный OAuth-поток недоступен — передайте `--api-key`. CLI проверяет
ключ, сохраняет его в `~/.rox/config.json` и очищает любую предыдущую
OAuth-сессию.

```bash
rox auth login --api-key sk_live_…
rox auth login --api-key sk_live_… --organization acme
```

<CommandReturns>
Только что аутентифицированный пользователь и активная организация.

<HumanOutput>
```text
◇  Authorized!
│  Satya Patel (you@example.com)
│  Organization: Acme
└  Logged in successfully.
```
</HumanOutput>

<JsonOutput>
```ts
{
  userId: string;
  organizationId: string;
  organizationName: string;
}
```
</JsonOutput>
</CommandReturns>
</Command>

<Command name="rox auth logout">
Очистить сохранённые учётные данные — как OAuth-сессию, так и сохранённый API-ключ.
Не вызывает API и не очищает активную организацию (ваша
предпочитаемая организация сохраняется между повторными входами).

```bash
rox auth logout
```

<CommandReturns>
Сообщение с подтверждением.

<HumanOutput>
```text
Logged out.
```
</HumanOutput>

<JsonOutput>
```ts
{ message: "Logged out." }
```
</JsonOutput>
</CommandReturns>
</Command>

<Command name="rox auth whoami">
Показать текущего пользователя, активную организацию и источник аутентификации.

```bash
rox auth whoami
```

<CommandReturns>
Текущий пользователь и контекст организации.

<HumanOutput>
```text

Organization: Acme
Auth: Session (expires in 32 min)
```
</HumanOutput>

<JsonOutput>
```ts
{
  userId: string;
  email: string;
  name: string;
  organizationId: string;
  organizationName: string;
  authSource: "override" | "config" | "oauth";
}
```
</JsonOutput>
</CommandReturns>
</Command>

### start

Запустить локальный хост-сервер — демон (daemon), с которым взаимодействуют
и CLI, и десктоп-приложение при работе с этой машиной.

<Command
	name="rox start"
	options={[
		{ flag: "--daemon", description: "Запустить в отсоединённом режиме." },
		{ flag: "--port <n>", description: "Конкретный порт. По умолчанию: свободный loopback-порт." },
	]}
>
Если манифест существует и PID жив, возвращает данные существующего
экземпляра. Иначе запускает бинарник хост-сервера, опрашивает
`/trpc/health.check` до 10 секунд и записывает манифест.
Привязывается только к `127.0.0.1`.

```bash
rox start --daemon
```

<CommandReturns>
Работающий хост-сервер.

<JsonOutput>
```ts
{
  pid: number;
  port: number;
  organizationId: string;
}
```
</JsonOutput>
</CommandReturns>
</Command>

### stop

Остановить локальный хост-сервер.

<Command name="rox stop">
Отправляет SIGTERM, ждёт до 10 секунд, отправляет SIGKILL, если процесс всё ещё жив.
Манифест удаляется во всех случаях — в том числе когда сам SIGTERM выбрасывает ошибку.

```bash
rox stop
```

<CommandReturns>
Остановленный хост-сервер, если он был запущен.

<JsonOutput>
```ts
// No manifest
{ running: false }

// Manifest existed
{ pid: number; organizationId: string }
```
</JsonOutput>
</CommandReturns>
</Command>

### status

Проверить состояние локального хост-сервера.

<Command name="rox status">
`healthy` отражает живой запрос `health.check` (таймаут 2 секунды).

```bash
rox status
```

<CommandReturns>
Состояние хост-сервера. Три формы в зависимости от того, что запущено.

<JsonOutput>
```ts
// No manifest
{ running: false; organizationId: string }

// Manifest exists but PID is dead
{
  running: false;
  stale: true;
  pid: number;
  organizationId: string;
}

// Running
{
  running: true;
  healthy: boolean;
  pid: number;
  port: number;
  endpoint: string;
  organizationId: string;
  uptimeSec: number;
}
```
</JsonOutput>
</CommandReturns>
</Command>

### update

Обновить Rox CLI и бинарник хост-сервера.

<Command
	name="rox update"
	options={[
		{ flag: "--check", description: "Только проверить наличие обновлений; не устанавливать." },
		{ flag: "--force", description: "Переустановить, даже если эта версия уже установлена." },
		{ flag: "--version <version>", description: "Установить конкретную версию (например, `0.1.2`) вместо последней по принципу rolling. Допускает обновление или откат." },
	]}
>
Загружает подходящий `rox-<platform>.tar.gz` из GitHub Releases
и атомарно заменяет корень установки. Доступно только в собранных
бинарниках; запуск из `bun run dev` завершается ошибкой.

```bash
rox update                     # upgrade to the latest release
rox update --check             # show current → target without installing
rox update --version 0.1.2     # install a specific version (up or down)
```

<CommandReturns>
Текущая и целевая версия, а также информация о том, изменилось ли что-либо.

<HumanOutput>
```text
Updated 0.1.4 → 0.1.5 (/Users/you/.local/share/rox)
```
</HumanOutput>

<JsonOutput>
```ts
// --check
{
  current: string;
  target: string;
  upToDate: boolean;
  pinned: boolean;
}

// install
{
  current: string;
  target: string;
  updated: boolean;
  installRoot?: string;
}
```
</JsonOutput>
</CommandReturns>
</Command>

### projects

Проект — это репозиторий, зарегистрированный в вашей организации. Он содержит
облачную запись (одну на репозиторий, общую для организации) и одну или несколько
host-side копий. Рабочие пространства ответвляются от копии на хосте.

<Command
	name="rox projects list"
	output={`Array<{
  id: string;
  name: string;
  slug: string;
  repoCloneUrl: string | null;
  githubRepositoryId: string | null;
}>`}
	humanColumns={["NAME", "SLUG", "REPO", "ID"]}
	quiet="project IDs"
>
Список проектов в активной организации. В рамках всей организации; не ограничен хостом.
</Command>

<Command
	name="rox projects create"
	options={[
		{ flag: "--name <name>", required: true, description: "Отображаемое имя проекта." },
		{ flag: "--host <id>", description: "Целевой machineId хоста. Взаимоисключающий с `--local`; один из них обязателен." },
		{ flag: "--local", description: "Целевая — эта машина. Взаимоисключающий с `--host`; один из них обязателен." },
		{ flag: "--clone <url>", description: "Клонировать из Git URL. Взаимоисключающий с `--import`." },
		{ flag: "--parent-dir <path>", description: "Родительский каталог, в который попадёт склонированный репозиторий. Обязателен с `--clone`." },
		{ flag: "--import <path>", description: "Путь к существующему репозиторию на целевом хосте. Взаимоисключающий с `--clone`." },
	]}
	output={`{
  projectId: string;
  repoPath: string;
  mainWorkspaceId: string;
}`}
>
Создать новый проект на хосте. Два режима:

- **Clone** — передайте `--clone <url>` и `--parent-dir`. Хост клонирует
  репозиторий в `<parent-dir>/<derived-name>/`.
- **Import** — передайте `--import <path>`, чтобы зарегистрировать репозиторий, который уже существует
  на диске.

В любом случае создаётся новая облачная запись проекта и обеспечивается наличие основного
рабочего пространства хоста.

`create` всегда создаёт новый облачный проект, даже если в вашей организации уже есть
проект для того же Git URL. Используйте [`projects setup`](#rox-projects-setup-id),
когда подключаете существующий проект на новой машине.

```bash
rox projects create --name "my-app" --local --clone https://github.com/org/my-app.git --parent-dir ~/code
rox projects create --name "my-app" --local --import ~/code/my-app
```
</Command>

<Command
	name="rox projects setup <id>"
	args={[
		{ name: "id", required: true, description: "UUID проекта для подключения." },
	]}
	options={[
		{ flag: "--host <id>", description: "Целевой machineId хоста. Взаимоисключающий с `--local`; один из них обязателен." },
		{ flag: "--local", description: "Целевая — эта машина. Взаимоисключающий с `--host`; один из них обязателен." },
		{ flag: "--parent-dir <path>", description: "Родительский каталог для клонирования репозитория проекта (режим clone). URL берётся из облачного проекта, поэтому `--clone <url>` не нужен." },
		{ flag: "--import <path>", description: "Путь к существующему репозиторию на целевом хосте (режим import)." },
		{ flag: "--allow-relocate", description: "Разрешить повторный импорт по другому пути, если проект уже настроен здесь. Только для `--import`." },
	]}
	output={`{
  repoPath: string;
  mainWorkspaceId: string | null;
}`}
>
Подключить существующий проект к хосту без создания дублирующей облачной
записи. Аналог [`create`](#rox-projects-create) для случая
«новая машина, репозиторий уже в организации».

Найдите ID проекта через [`projects list`](#rox-projects-list).

Идемпотентно: повторный запуск для проекта, который уже настроен по тому же
пути, не выполняет никаких действий и возвращает существующий `mainWorkspaceId`.

```bash
# Pick an ID from `rox projects list`, then clone it onto this machine
rox projects setup 47c31b04-… --local --parent-dir ~/code

# Or register an existing local checkout
rox projects setup 47c31b04-… --local --import ~/code/my-app
```
</Command>

Обнаружение хостов, зарегистрированных в активной организации. Чтобы управлять
хост-сервером, работающим на этой машине, используйте [`start`](#start),

<Command
	name="rox hosts list"

### workspaces (alias: `ws`)

Рабочие пространства — это рабочие копии на хосте, ограниченные веткой.

**Маршрутизация:** когда `--host` разрешается в локальную машину, CLI обращается к
хост-серверу напрямую через loopback HTTP — без обращения к облаку, работает
офлайн. Иначе запрос маршрутизируется через облачный API и реле.

Если разрешённый хост — локальная машина, но хост-сервер не
отвечает, CLI завершается с ошибкой и указывает на `rox start`, а не
молча обращается к облаку.

<Command
	name="rox workspaces list"
	options={[
		{ flag: "--host <id>", description: "Фильтровать по конкретному machineId хоста." },
		{ flag: "--local", description: "Фильтровать по этой машине." },
	]}
	output={`Array<{
  id: string;
  name: string;
  branch: string;
  projectId: string;
  projectName: string;
  hostId: string;
  hostName: string;
}>`}
	humanColumns={["NAME", "BRANCH", "PROJECT", "HOST"]}
	quiet="workspace IDs"
>
Список рабочих пространств, доступных активной организации. Без указания
`--host` или `--local` список действует в рамках всей организации.
</Command>

<Command
	name="rox workspaces create"
	options={[

		{ flag: "--branch <branch>", description: "Ветка рабочего пространства. Обязательна, если не задан `--pr`." },
		{ flag: "--pr <number>", description: "Номер pull request. Извлекает проверенный head PR." },
		{ flag: "--base-branch <branch>", description: "Ветка, от которой ответвляться, когда `--branch` не существует. По умолчанию — ветка проекта по умолчанию." },
		{ flag: "--host <id>", description: "Целевой machineId хоста. Взаимоисключающий с `--local`; один из них обязателен." },
		{ flag: "--local", description: "Целевая — эта машина. Взаимоисключающий с `--host`; один из них обязателен." },
		{ flag: "--agent <preset|uuid|rox>", description: "Агент для запуска после создания." },
		{ flag: "--prompt <text>", description: "Начальный промпт. Обязателен, когда задан `--agent`." },
		{ flag: "--command <cmd>", description: "Shell-команда для запуска в новом рабочем пространстве после создания." },
		{ flag: "--attachment <path>", description: "Локальный файл для загрузки для запущенного агента. Повторяемый." },

Создать рабочее пространство на целевом хосте. Укажите ровно один из `--branch`
или `--pr`. Если вы передаёте `--agent`, вы также должны передать `--prompt`. `--command`
выполняет одноразовую shell-команду в worktree и не зависит от
`--agent`/`--prompt` — передайте любой из них или оба.

```bash
rox workspaces create \
  --project prj_… \
  --name "fix-login-bug" \
  --branch fix/login-bug \
  --local

rox workspaces create \
  --project prj_… \
  --name "review-pr-123" \
  --pr 123 \
  --local
```
</Command>

### agents

Агенты — это записи терминальных агентов, настроенные на хосте (те же записи, что показаны в
Settings → Agents на этой машине). Каждая запись хранит `command`, `args`,
режим транспорта промпта и окружение, используемые для запуска агента в новой
сессии терминала внутри рабочего пространства.

<Command
	name="rox agents list"
	options={[
		{ flag: "--host <id>", description: "Целевой machineId хоста." },
		{ flag: "--local", description: "Целевая — эта машина. Взаимоисключающий с --host; ровно один обязателен." },
	]}
	output={`Array<{
  id: string;
  presetId: string;
  label: string;
  command: string;
  args: string[];
  promptTransport: "argv" | "stdin";
  promptArgs: string[];
  env: Record<string, string>;
  order: number;
}>`}
	humanColumns={["LABEL", "PRESET", "COMMAND", "ID"]}
	quiet="agent instance IDs"
>
Список агентов, настроенных на хосте. Первый вызов на новом хосте создаёт
встроенные значения по умолчанию.

```bash
rox agents list --local
rox agents list --host host_…
```
</Command>

<Command
	name="rox agents create"

## Режимы вывода

**Режим JSON** (`--json`): сырые полезные нагрузки. Списки выводят массивы, get/create/update
выводят объекты, delete выводит объекты-сводки. Без обёртки `{ "data": ... }`.
Пустые результаты выводят `null`.

**Тихий режим** (`--quiet`): только ID. Массивы объектов с полем `id`
выводят по одному ID на строку; одиночные объекты выводят свой `id`; всё остальное
откатывается к JSON.

Когда `CLAUDE_CODE`, `CLAUDECODE`, `CLAUDE_CODE_ENTRYPOINT`, `CODEX_CLI`,
`GEMINI_CLI`, `ROX_AGENT` или `CI` установлены в непустое значение, вывод
по умолчанию — JSON, если только не передан `--quiet`.
