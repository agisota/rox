Команда `rox start` запускает headless HTTP-сервер, с которым может взаимодействовать любое десктоп-приложение Rox или CLI.

Он управляет рабочими пространствами, портами и запусками агентов на машине, где работает. Десктоп-приложение запускает его автоматически; на машине только с CLI или headless-машине вы запускаете его сами.

Если вы установили десктоп-приложение, оно уже включает CLI и устанавливает
shim `~/.rox/bin/rox` при запуске приложения. Терминалы Rox автоматически
добавляют этот каталог в начало `PATH`. На headless-машине используйте
автономный установщик из раздела [Начало работы](/cli/getting-started).

## Предварительные требования

`git` и `gh` должны быть в вашем `PATH`. `git` обрабатывает операции clone, worktree и работу с ветками; `gh` обрабатывает checkout PR и метаданные.

```bash
gh auth login
```

Хост-сервер повторно использует вашу существующую аутентификацию `gh`. Для песочниц или CI задайте `GH_TOKEN` (или `GITHUB_TOKEN`) вместо запуска `gh auth login` — `gh` подхватит его автоматически.

### Агентские CLI (опционально)

Если вы планируете запускать AI-агентов внутри рабочих пространств, установите те агентские CLI, которые хотите иметь доступными. Хост-сервер запускает их в зависимости от того, какие из них находятся в вашем `PATH`:

| Агент | Установка |
|-------|---------|
| Claude Code | `npm install -g @anthropic-ai/claude-code` |
| Codex | `npm install -g @openai/codex` |
| Cursor Agent | установка через сайт Cursor |
| Amp | `npm install -g @sourcegraph/amp` |
| Gemini CLI | `npm install -g @google/gemini-cli` |
| OpenCode | см. [opencode.ai/docs](https://opencode.ai/docs) |

Ни один из них не требуется для запуска хост-сервера или создания рабочих пространств — устанавливайте только то, что планируете запускать.

## Аутентификация Rox CLI

Хост-сервер запускается под вашей сессией Rox. Если вы ещё не вошли:

```bash
rox auth login
```

См. [Начало работы](/cli/getting-started) для полного процесса входа,
включая аутентификацию по API-ключу для CI.

## Запуск хост-сервера

```bash
rox start --daemon
```

Запускает демон (daemon) в фоне и записывает манифест по пути
`~/.rox/host/<organizationId>/manifest.json`, чтобы другие клиенты могли его найти.
Привязывается только к `127.0.0.1`.

Проверьте, что он работает корректно:

```bash
rox status
```

## Начало работы

### Создание или подключение проекта

Рабочее пространство ответвляется от проекта — репозитория, зарегистрированного в Rox.

Каждый репозиторий получает одну облачную запись проекта, общую для всей организации.
Первая машина, регистрирующая репозиторий, запускает `projects create`; каждая
машина после этого использует `projects setup`, чтобы подключиться к существующему
проекту вместо создания дубликата.

Начните с проверки того, что уже есть в вашей организации:

```bash
rox projects list
```

#### Первая машина — `projects create`

Если проект ещё не существует, склонируйте удалённый репозиторий или импортируйте
существующий локальный:

```bash
# Клонировать из Git URL
rox projects create \
  --name "My Project" \
  --local \
  --clone https://github.com/org/repo.git \
  --parent-dir ~/code

# Или импортировать репозиторий, который уже есть на диске
rox projects create \
  --name "My Project" \
  --local \
  --import ~/code/existing-repo
```

Хост-сервер клонирует (или регистрирует) репозиторий, создаёт облачную запись
проекта и обеспечивает наличие основного рабочего пространства.

#### Последующие машины — `projects setup <id>`

Если коллега (или ваша предыдущая машина) уже создал проект,
подключитесь к нему вместо повторного запуска `create` — `create` создал бы дубликат
облачного проекта, указывающего на тот же репозиторий. Выберите ID из `projects list`,
затем:

```bash
# Клонировать репозиторий проекта на эту машину
rox projects setup 47c31b04-… --local --parent-dir ~/code

# Или подключиться к существующей локальной копии
rox projects setup 47c31b04-… --local --import ~/code/repo
```

URL для режима клонирования берётся из облачной записи проекта, поэтому `setup`
не требует `--clone <url>`. `setup` идемпотентен: повторный запуск для
проекта, который уже настроен здесь, не выполняет никаких действий.

### Создание первого рабочего пространства

```bash
rox projects list
rox workspaces create \
  --project prj_… \
  --name "fix-login-bug" \
  --branch fix/login-bug
```

Хост-сервер клонирует проект (если его ещё нет на диске), создаёт
git worktree на выбранной ветке и регистрирует рабочее пространство.

Вы также можете создать рабочее пространство по номеру pull request:

```bash
rox workspaces create \
  --project prj_… \
  --name "review-pr-123" \
  --pr 123 \
  --local
```

## Остановка хост-сервера

```bash
rox stop
```

## Общее состояние

Состояние хранится в `~/.rox/host/<organizationId>/`:

```text
~/.rox/host/<organizationId>/manifest.json    # endpoint + auth token
~/.rox/host/<organizationId>/host.db          # local host server DB
```

Хост-сервер, запущенный десктоп-приложением, виден для CLI и
наоборот. `ROX_HOME_DIR` перемещает всё дерево целиком.
