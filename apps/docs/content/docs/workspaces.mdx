## Основная концепция

Каждое рабочее пространство = одна ветка git. Рабочие пространства — это изолированные git worktrees с собственными директорией, терминалом и портами.

## Создание рабочих пространств

Нажмите **⌘N** или кликните «New Workspace». Создавайте из:

- **New branch** — новая ветка от main
- **Existing branch** — продолжение работы над веткой
- **Pull request** — просмотр или продолжение PR

В сценарии создания рабочего пространства «сначала промпт» опция **Link pull request** позволяет искать по заголовку или номеру PR либо вставить полный URL PR для прямого сопоставления.

Каждое рабочее пространство получает:
- Собственную ветку git
- Собственную рабочую директорию
- Собственные сессии терминала
- Обнаружение и управление портами в пределах рабочего пространства

## Импорт worktrees

Если у вас уже есть git worktrees на диске, используйте вкладку **Import** в модальном окне New Workspace, чтобы добавить их в Rox. Нажмите **Import all**, чтобы массово импортировать все обнаруженные worktrees сразу.

## Статус Ahead/Behind

Для **локальных рабочих пространств (на основе веток)** боковая панель показывает, насколько активная ветка опережает или отстаёт от своей удалённой ветки:

- **↑N** — на N коммитов впереди удалённой ветки (не отправлены)
- **↓N** — на N коммитов позади удалённой ветки (требуется pull/rebase)

## Статус PR

Рабочие пространства на основе PR показывают состояние pull request в боковой панели.

Наведите курсор на рабочее пространство, чтобы увидеть статус ревью, проверки CI и ссылки GitHub. Если у ветки есть деплой-превью, вы также увидите кнопку **Open Preview**.

## Сценарии использования

- **Параллельные фичи** — работа над несколькими функциями одновременно
- **AI-агенты** — запуск Claude Code в отдельных рабочих пространствах
- **Код-ревью** — просмотр PR без остановки текущей работы

## Рекомендации

- Используйте описательные имена веток: `feature/auth`, `fix/login-bug`
- Не работайте напрямую в `main`
- Удаляйте рабочие пространства после мержа
