## Обзор

Автоматизации запускают сессию агента для проекта по расписанию — как cron-задача, но результатом становится живое рабочее пространство, которое вы (или коллега) можете открыть, просмотреть и продолжить интерактивно.

Типичные сценарии:

- **Standup'ы** — ночная сводка по PR, задачам и активности
- **Release notes** — генерация черновика из смерженных коммитов
- **Аудиты** — ежеутренний поиск проблем безопасности или устаревших API
- **Проверки зависимостей** — еженедельная проверка устаревших пакетов

## Требования

- **Проект v2** — автоматизации запускаются для проекта, уже связанного с репозиторием GitHub.
- **Целевое устройство** — запуск происходит на устройстве, которое вы выбираете при создании автоматизации.

## Создание автоматизации

Нажмите **Automations → New automation** и заполните:

- **Title** — как автоматизация отображается в списке
- **Prompt** — инструкции, которые агент получит в момент запуска. Поддерживает markdown, упоминания `@file/path` (в пределах выбранного проекта), коды `:emoji:` и slash-команды `/`.
- **Device** — какой хост запускает сессию. По умолчанию это устройство.
- **Project** — проект v2, для которого запускается сессия.
- **Schedule** — выберите пресет или введите собственный RRule. Все времена указываются в вашем местном часовом поясе.
- **Agent** — Claude, Codex, Amp и т. д. — всё, что вы настроили в **Settings → Agents**.

## Расписания

Расписания хранятся в формате [RFC 5545 RRules](https://datatracker.ietf.org/doc/html/rfc5545#section-3.8.5). В выборе расписания есть пресеты для распространённых периодичностей; для всего остального используйте **Custom RRule…**.

Примеры:

- `FREQ=DAILY;BYHOUR=9;BYMINUTE=0` — каждый день в 9:00
- `FREQ=WEEKLY;BYDAY=MO,TU,WE,TH,FR;BYHOUR=9;BYMINUTE=0` — по будням в 9:00
- `FREQ=WEEKLY;BYDAY=FR;BYHOUR=17;BYMINUTE=0` — каждую пятницу в 17:00

## Запуски

Каждый раз, когда срабатывает автоматизация, записывается **запуск**. Запуски отображаются на странице автоматизации со статусом:

- **dispatched** — сессия успешно передана целевому устройству
- **skipped (offline)** — целевое устройство не было в сети в момент срабатывания; запуск пропущен, а расписание сдвинулось дальше
- **dispatch failed** — что-то сломалось при передаче; проверьте сообщение об ошибке

Когда запуск происходит в режиме «новое рабочее пространство», для этого запуска создаётся свежее рабочее пространство из репозитория проекта. Откройте его из строки запуска, чтобы увидеть работу агента.

## Пауза, запуск сейчас, удаление

На странице автоматизации:

- **Pause** — останавливает планирование, пока вы не возобновите. Существующие запуски сохраняются.
- **Run now** — запускает немедленно. Запуск появляется в **Previous runs** в течение нескольких секунд.
- **Delete** — удаляет автоматизацию и историю её запусков.

## Ограничения

- **Доставка не менее одного раза** — в редких случаях автоматизации могут запуститься более одного раза (например, при повторных попытках диспетчера). Проектируйте промпты так, чтобы их повторный запуск был безопасным.
- **Нет отслеживания завершения (v1)** — запуски доходят до статуса `dispatched`; страница автоматизации не показывает успех или неудачу самого агента. Откройте рабочее пространство, чтобы увидеть, что произошло.
- **Офлайн-хосты пропускаются молча** — если ваше устройство офлайн в момент срабатывания, запуск помечается как `skipped_offline`, и следующее срабатывание планируется как обычно.
- **Один промпт на автоматизацию** — у каждой автоматизации один промпт. Чтобы запускать несколько промптов по расписанию, создайте несколько автоматизаций.

## CLI

```bash
rox automations list
rox automations create --name "Nightly audit" --rrule "FREQ=DAILY;BYHOUR=3;BYMINUTE=0" --project <projectId> --prompt-file prompt.md --agent claude
rox automations get <id>
rox automations run <id>
rox automations pause <id>
rox automations resume <id>
rox automations delete <id>
```
