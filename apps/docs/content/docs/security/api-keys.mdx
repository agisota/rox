## Зачем нужны API-ключи

API-ключ даёт программный доступ к Rox без интерактивного входа через OAuth. Ключи используются в:

- **CLI** — через `ROX_API_KEY` или флаг `--api-key`;
- **TypeScript SDK** — при инициализации клиента;
- **MCP-сервере** — для подключения внешних агентов;
- **прямых REST/tRPC-вызовах** — как Bearer-токен.

Ключи имеют префикс `sk_live_` и выпускаются на уровне better-auth (плагин `apiKey`) с привязкой к активной организации.

## Создание ключа

Ключ создаётся в контексте **активной организации**. Через tRPC это процедура `apiKey.create`:

```ts
const { key } = await trpc.apiKey.create.mutate({ name: "ci-runner" });
// key возвращается ОДИН раз — сохраните его сразу.
```

<Callout type="warn">
Полное значение ключа (`sk_live_…`) возвращается только в момент создания. Сохраните его сразу в безопасном месте (менеджер секретов, переменные окружения CI). Повторно посмотреть его нельзя — потерянный ключ нужно отозвать и выпустить заново.
</Callout>

Если у сессии нет активной организации, создание ключа отклоняется — сначала выберите организацию.

## Использование ключа

<Tabs items={["CLI", "Переменная окружения", "Bearer-заголовок"]}>
<Tab value="CLI">

```bash
rox --api-key sk_live_xxx status
```

</Tab>
<Tab value="Переменная окружения">

```bash
export ROX_API_KEY=sk_live_xxx
rox status
```

</Tab>
<Tab value="Bearer-заголовок">

```bash
curl https://api.rox.one/api/trpc/... \
  -H "Authorization: Bearer sk_live_xxx"
```

</Tab>
</Tabs>

## Хранение и гигиена секретов

- Никогда не коммитьте ключи в репозиторий и не вставляйте их в публичные логи.
- Храните ключи в менеджере секретов или в переменных окружения CI.
- Выпускайте отдельный ключ под каждую интеграцию (`name`), чтобы можно было отозвать один, не затронув остальные.
- При компрометации немедленно отзывайте ключ и выпускайте новый.
