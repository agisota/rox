## Обзор

Аутентификация в Rox построена на **better-auth**. Бэкенд (`apps/api`) — единый источник истины для входа: к нему обращаются веб, десктоп, мобильное приложение, CLI и SDK.

## Методы входа

| Метод | Где доступен | Примечание |
| --- | --- | --- |
| **GitHub OAuth** | Прод и dev | Основной способ входа для пользователей. |
| **Magic link (приглашения)** | Прод и dev | Подписанный одноразовый токен в письме-приглашении в организацию. |
| **Email + пароль / Local Admin (dev)** | Только `NODE_ENV=development` | Кнопка «Sign in as dev» и пара `admin@local.test` для локальной разработки. В продакшене отключены. |

<Callout type="warn">
Email/пароль намеренно включается только в режиме разработки. Включение его в продакшене — это отдельное продуктовое и security-решение: сначала нужны верификация email и rate-limiting.
</Callout>

## Сессии

- Сессии хранятся в базе данных и живут **30 дней** (`expiresIn`), с обновлением активности раз в сутки (`updateAge`).
- Включён кэш сессии в cookie на короткий срок (5 минут), чтобы снизить число обращений к БД.
- Cookie работают через поддомены (`crossSubDomainCookies`) на общем домене, так что `app.rox.one`, `api.rox.one` и другие поверхности делят одну сессию.
- Доверенные источники (`trustedOrigins`) включают веб, API, marketing, admin, опционально desktop, схемы `rox://` для десктопа и `exp://` для Expo в dev.

## Организации, участники и роли

Rox использует плагин **organization** better-auth:

- Создатель организации получает роль **owner**.
- Каждый участник принадлежит хотя бы одной **команде** (team); при создании организации заводится команда по умолчанию, новые участники автоматически добавляются в старейшую команду.
- Активная организация сессии (`activeOrganizationId`) определяет контекст всех запросов; членства пользователя (`organizationIds`) прокидываются в токены.
- Приглашения живут 7 дней и ограничены rate-лимитом (**не более 10 приглашений в час** на приглашающего). Право пригласить роль проверяется функцией `canInvite`.
