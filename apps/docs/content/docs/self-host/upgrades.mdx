## Обзор

Обновление self-host Rox — это три шага: подтянуть код, применить миграции схемы, пересобрать и перезапустить сервисы. Применение миграций — единственный потенциально необратимый шаг, поэтому к нему относимся осторожно.

## Шаг 1 — Подтянуть код

```bash
git fetch origin
git checkout main
git pull
bun install
```

После обновления зависимостей всегда переустанавливайте их (`bun install`) — могли смениться версии пакетов.

## Шаг 2 — Применить миграции

Новые миграции лежат в `packages/db/drizzle/` (генерируются автоматически, не редактируются вручную).

<Callout type="warn">
Применение миграций (`drizzle-kit migrate` / `push`) — это шаг деплоя против реальной БД. Сначала сделайте резервную копию/снапшот БД и при возможности прогоните миграцию на ветке-копии (например, на отдельной ветке Neon), прежде чем применять к продакшену.
</Callout>

Рекомендуемый порядок:

1. Снимите бэкап/снапшот текущей БД.
2. Прогоните миграцию на копии данных и проверьте, что приложение поднимается.
3. Примените миграцию к целевому окружению через явный, человеко-гейтированный процесс деплоя.

При локальной разработке достаточно перезапустить идемпотентный скрипт настройки — он применит новые миграции:

```bash
./.rox/setup.local.sh
```

## Шаг 3 — Пересборка и перезапуск

```bash
bun run build
bun run dev   # или перезапуск ваших продакшен-процессов
```

На Render автодеплой из `main` выполняет сборку и перезапуск автоматически (см. [Render](/self-host/render)).

## Откат

- **Код:** вернитесь на предыдущий тег/коммит и пересоберите.
- **База данных:** откат миграций требует восстановления из снапшота, снятого на шаге 2, либо forward-миграции, отменяющей изменения. Именно поэтому бэкап перед применением обязателен — простого «отката миграции» против продакшена без снапшота нет.
