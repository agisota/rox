## Обзор

Выполняйте команды автоматически при создании или удалении рабочих пространств и определяйте команды dev-сервера, которые запускаются через кнопку Run.

Создайте `.rox/config.json` в вашем проекте:

```json
{
  "setup": ["bun install", "cp \"$ROX_ROOT_PATH/.env\" .env"],
  "teardown": ["docker-compose down"],
  "run": ["./.rox/run.sh"]
}
```

## Как это работает

1. Создание рабочего пространства → команды **setup** выполняются в терминале
2. Удаление рабочего пространства → выполняются команды **teardown**
3. Нажатие кнопки **Run** → команды **run** выполняются в отдельной панели

Команды setup и teardown выполняются последовательно в каталоге рабочего пространства.

## Скрипт запуска

Поле `run` определяет команды, которые запускают ваш dev-сервер или другие долго работающие процессы. В отличие от setup/teardown, команды run:

- **По запросу** — запускаются кнопкой Run, а не автоматически при создании рабочего пространства
- **Перезапускаемые** — останавливаются и перезапускаются из интерфейса без пересоздания рабочего пространства
- **Отдельная панель** — выполняются в собственной панели терминала, отдельной от вашей оболочки

```json
{
  "setup": ["npm install"],
  "run": ["./.rox/run.sh"]
}
```

## Переменные окружения

| Переменная | Описание |
|----------|-------------|
| `ROX_ROOT_PATH` | Путь к корневому репозиторию |
| `ROX_WORKSPACE_NAME` | Имя текущего рабочего пространства |
| `ROX_WORKSPACE_PATH` | Путь к worktree рабочего пространства |

## Примеры

**Node.js:**
```json
{ "setup": ["bun install", "cp \"$ROX_ROOT_PATH/.env\" .env"] }
```

**Docker:**
```json
{
  "setup": ["docker-compose up -d", "bun run db:migrate"],
  "teardown": ["docker-compose down -v"]
}
```

**Полная конфигурация:**
```json
{
  "setup": ["./.rox/setup.sh"],
  "teardown": ["./.rox/teardown.sh"],
  "run": ["./.rox/run.sh"]
}
```

## Пользовательские переопределения

Переопределяйте скрипты проекта без изменения репозитория, разместив `config.json` в вашем домашнем каталоге:

```
~/.rox/projects/<project-id>/config.json
```

Где `<project-id>` — это уникальный ID вашего проекта в Rox (виден в настройках проекта в приложении).

### Порядок приоритета

Конфигурация разрешается в этом порядке (побеждает первая найденная, для setup, teardown и run):
1. `~/.rox/projects/<project-id>/config.json` — пользовательское переопределение
2. `<worktree>/.rox/config.json` — специфичная для worktree
3. `<repo>/.rox/config.json` — по умолчанию для проекта

Слияние между уровнями не происходит — целиком используется первая найденная конфигурация.

### Примеры

**Пользовательский скрипт инициализации:**
```json
{
  "setup": ["~/.rox/projects/<project-id>/setup.sh"],
  "teardown": ["~/.rox/projects/<project-id>/teardown.sh"]
}
```

**Полностью пропустить инициализацию:**
```json
{ "setup": [], "teardown": [] }
```

## Локальная конфигурация

Расширяйте или переопределяйте закоммиченные командой скрипты без изменения репозитория. Создайте `.rox/config.local.json` рядом с `config.json` — он автоматически добавляется в gitignore.

### Добавление в начало и/или в конец

Добавляйте шаги до или после скриптов команды:

```json
{
  "setup": {
    "before": ["echo 'running pre-setup'"],
    "after": ["./.rox/my-post-setup.sh"]
  },
  "teardown": {
    "after": ["./.rox/my-cleanup.sh"]
  }
}
```

Результат: `before` → закоммиченные скрипты setup → `after`. Ключи, которые вы не указываете, проходят без изменений.

### Полное переопределение

Используйте обычный массив, чтобы полностью заменить скрипты команды:

```json
{
  "setup": ["./.rox/my-custom-setup.sh"]
}
```

### Сочетание подходов

Переопределите один ключ, расширяя другой:

```json
{
  "setup": { "after": ["bun run my-extra-step"] },
  "teardown": ["my-custom-teardown.sh"]
}
```

### Порядок разрешения

Локальная конфигурация накладывается поверх той базовой конфигурации, которая побеждает в [порядке приоритета](#priority-order). Если `config.local.json` существует и в worktree, и в основном репозитории, приоритет имеет локальная конфигурация worktree.

## Принудительное удаление

Если скрипты teardown завершаются с ошибкой при удалении рабочего пространства, вы можете выполнить принудительное удаление, чтобы пропустить неудавшийся teardown и продолжить очистку.

При сбое teardown:
1. Появляется уведомление об ошибке с кнопками «Delete Anyway» и «View Logs»
2. Нажмите «Delete Anyway», чтобы немедленно выполнить принудительное удаление
3. Или нажмите «View Logs», чтобы изучить сбой, а затем выполните принудительное удаление из диалога

## Советы

- Делайте инициализацию быстрой — она выполняется при каждом создании рабочего пространства
- Закоммитьте `.rox/`, чтобы поделиться с командой
- Используйте shell-скрипты для сложной логики: `"setup": ["./.rox/setup.sh"]`
- Используйте `config.local.json`, чтобы добавить личные шаги, не трогая конфигурацию команды
- Используйте пользовательские переопределения (`~/.rox/projects/`), чтобы полностью заменить скрипты для каждого проекта
- Используйте `run` для dev-серверов вместо того, чтобы помещать их в `setup` — скрипты run перезапускаемы и не блокируют создание рабочего пространства
